# UI Style Guide

## 1. Purpose
This document defines the visual and linguistic standards for NexArtWO screens, components, and documents. The primary goal is to maintain a clean, professional, and business-focused user experience that avoids technical jargon and maintains strict consistency.

## 2. Visual Principles
- **Cleanliness:** Interfaces must not be overloaded. Use whitespace effectively.
- **Clarity:** Cards and sections must be clearly delineated.
- **Simplicity:** Keep the cognitive load low for business users.
- **Consistency:** Similar actions and components should look identical across different tabs/modules.

## 3. Tone of Visible Text
- Use simple, direct, business-appropriate language.
- Expose zero technical jargon to the end-user. Do not use words like `schema`, `namespace`, `RLS`, `backend`, or `migration`.

## 4. Buttons
- Maintain consistent styles (`btn-primary`, `btn-secondary`, `btn-danger`, etc.).
- Align buttons logically (e.g., Cancel on the left, primary Action on the right).

## 5. Inputs
- Form inputs must have clear visual borders and padding (`form-control`).
- Labels must be legible, typically smaller, bold, and in a secondary text color.

## 6. Modals
- Must be grouped by logical sections (e.g., Core Info vs. Financial Setup).
- Use internal scrolling for tall modals on small screens.
- Avoid overwhelming the user; space elements appropriately.

## 7. Cards
- Use cards to segment distinct pieces of information or entities (e.g., Project cards).
- Maintain consistent borders, padding, and hover states.

## 8. Tables
- Simple, border-bottom separated rows.
- Clear, capitalized, muted headers.

## 9. Badges
- Used for status indicators (e.g., Active, Pending).
- Colors must map logically to the status (e.g., Green for confirmed/active, Amber for pending).

## 10. Empty States
- Empty states must be instructive and explicitly state what features are unlocked by taking the primary action.
- Must include a clear Call-To-Action (CTA) button.

## 11. Sidebar Labels
- Must describe the destination contextually.
- Avoid internal developer milestones (e.g., "Sprint 2").

## 12. Internal/Admin Banners
- Features intended for internal tracking or admin use only (e.g., Investor Hub) must have a clear, visually distinct banner (often with a lock icon) denoting restricted use.

## 13. Document Style Consistency
- Documentation files must follow a similar visual and structural line according to their purpose (e.g., standardized headers, bullet lists, markdown tables).

## 14. Examples of Good Labels
- "Inside Project"
- "Create First Project"
- "Financial Setup"
- "Internal financial tracking"
- "Coming Soon"
- "Admin Only"

## 15. Examples of Labels to Avoid
- "Sprint 2"
- "Sprint 3"
- "schema"
- "namespace"
- "RLS"
- "backend"
- "migration"
